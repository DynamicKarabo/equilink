Prompt 1: Core Domain & State Machine Implementation Implement the OrderAggregate and its finite state machine using the Stateless library. The aggregate must hold no mutable state fields; its state must be entirely derived by replaying a domain event stream. Implement the specific Phase 1 lifecycle transitions: New → RiskValidating → Approved/Rejected → Submitted. Ensure that no code mutates order state directly; every valid transition trigger must explicitly fire a persisted domain event.
Prompt 2: Idempotent Gateway Pipeline Build the IdempotencyBehavior for the MediatR pipeline. Enforce order deduplication atomically using Redis SET NX with a 24-hour TTL. Rely on caller-generated idempotency keys (UUID v4) and scope these keys per fundId to prevent cross-tenant collisions. If a retry is detected, return the cached identical HTTP 200 response alongside the original orderId, rather than a 409 conflict.
Prompt 3: Pre-Trade Risk Engine Implement the pre-trade risk engine as a chain of responsibility hanging off the MediatR pipeline. Develop the SymbolBlacklistRule and MaxOrderSizeRule. These rules must evaluate limits exclusively against Redis-cached state. You must never query PostgreSQL synchronously on the hot execution path.
Prompt 4: Append-Only Event Store Infrastructure Create the PostgreSQL event store schema and Entity Framework Core 8 mappings for order_events. Configure the table to use JSONB for all event payloads and event type discriminators. You must enforce the append-only contract by applying database-level REVOKE UPDATE, DELETE privileges on the order_events table.
Prompt 5: Multi-Tenant Data Isolation Implement strict multi-tenant isolation at the Fund level using Entity Framework Core global query filters. Create an ICurrentFundContext populated from the JWT fund_id claim at the start of each request. Configure EF Core so that every query automatically appends a WHERE fund_id = @fundId clause, making cross-tenant data access impossible without explicit, auditable overrides.
Prompt 6: Read Models & API Endpoints Implement the POST /orders endpoint (configured for paper trading) and a GET /orders/{id} endpoint. Build the read path for the GET endpoint using Dapper to query an OrderSummaryProjection. Ensure the write operations (via EF Core) are fully separated from these high-performance Dapper read queries to satisfy the system's CQRS architecture
